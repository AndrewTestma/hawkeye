Docker基础(一):Linux Namespace
发表于 2020-07-21| 更新于 2020-07-21|Docker
字数总计:2.9k|阅读时长: 11 分钟
| 阅读量:|评论数:

Linux Namespace是Kernel的一个功能，是一种内核级别的资源隔离机制。该功能对内核资源进行分区，以使不同的进程看到不同的资源。资源可能存在于多个空间中。

概念

Linux Namespace起源于2002年，版本为2.4.19内核，涉及的是Mount Namespace。内核版本3.8中引入了User Namespace，从而为容器提供了足够的支持功能。从内核版本5.6开始，存在8种Namespace。

名称 系统调用参数 隔离的内容
Mount CLONE_NEWNS Mount points (since Linux 2.4.19)
IPC CLONE_NEWIPC System V IPC, POSIX message queues (since Linux 2.6.19)
UTS CLONE_NEWUTS Hostname and NIS domain name (since Linux 2.6.19)
Network CLONE_NEWNET network device interfaces, IPv4 and IPv6 protocol stacks, IP routing tables, firewall rules, the /proc/net and /sys/class/net directory trees, sockets, etc (since Linux 2.6.24)
PID CLONE_NEWPID Process IDs (since Linux 2.6.24)
User CLONE_NEWUSER User and group IDs (started in Linux 2.6.23 and completed in Linux 3.8)
Cgroup CLONE_NEWCGROUP Cgroup root directory (since Linux 4.6)
Time CLONE_NEWTIME System times (since Linux5.6)

内核中的每个命名空间为每个进程都分配了一个符号链接/proc/<pid>/ns/。此符号链接指向的索引节点号对于此命名空间中的每个进程都是相同的。通过readlink读取符号链接返回一个字符串，其中包含命名空间的种类和命名空间的索引节点编号。

Namespace的API主要使用如下3个系统调用：

  • clone():创建新进程
  • unshare():调用进程脱离某个命名空间(unshare命令)
  • setns():调用的进程加入一个已存在的命名空间

UTS Namespace

UTS namespace主要用来隔离hostname和NIS domain name两个系统标识。在UTS Namespace里面，每个Namespace允许有自己的hostname。拥有了独立的主机名和域名，在网络上可以被视为一个独立的节点，而并非宿主机上的一个进程。

IPC namespace

IPC namespace隔离的是IPC(Inter-Process Communication)资源，也就是进程间通信的方式(进程间通信采用的方式包括常见的信号量、消息队列和共享内存),包括System V IPC和POSIX message queues。只有同一个namespace下进程之间才能够通信。

下面这些/proc中内容对于每个namespace都是不同的：

  • /proc/sys/fs/mqueue下的POSIX message queues
  • /proc/sys/kernel下的 System V IPC
  • /proc/sysvipc保存了该namespace下的System V IPC 信息。

在Linux使用IPC，需要用到两个命令：

  • ipcs：查看IPC(共享内存、消息队列和信号量)的信息
  • ipcmk：创建IPC(共享内存、消息队列和信号量)的信息

PID Namespace


PID namespace 隔离的是进程pid属性，也就是说不同的namespace中的进程可以有相同的pid。每个PID namespace都有自己的计数程序。

PID namespace通过unshare和setns系统调用都不会把当前的进程加入到新的namespace，而把该进程的子进程进入到里面。因为pid是非常重要的信息，如何unshare或者setns把当前进程加入到新的namespace中，那么进程的pid将会发送变化，原来的pid也会被其它进程使用，会导致很多程序出现问题。

PID namespace是可以嵌套的。除了 root PID namespace 之外，所有的 PID namespace 都有一个父PID namespace，这个父PID namespace 就是执行 clone 或者 unshare 创建该 namespace 进程所在的 PID namespace。所属的父节点可以查看子节点中的进程，并可以通过信号量等方式影响子节点中的进程。

Mount namespace

Mount Namespace用来隔离各个进程看到的挂载点视图。在不同Namespace的进程中，看到得文件系统层次是不一样得。Namespace内的mount points可以通过mount(2)和umount(2)来修改，因此Mount可以用来实现文件系统的隔离。

/proc/<pid>/mounts可查看所有挂载在当前Namespace中的文件系统。
/proc/<pid>/mountstats可查看Namespace中文件设备的统计信息，包括挂载文件的名字、文件系统类型、挂载位置等。

进程在创建Mount Namespace时，会把当前的文件结构复制给新的namespace。新namespace中的mount操作都只影响自身的文件系统，而对外界不会产生任何影响。但是完全隔离在某些场景下并不适用，比如系统中插入新磁盘，虽然每个namespace 都重新mount一次也能达到效果，但是操作起来比较繁琐。所以在2006年的时候引入了挂载传播(mount propagation)。

挂载传播定义了挂载对象(mount object)之间的关系，系统用这些关系决定任何挂载对象中的挂载事件如何传播到其它挂载对象。

  • 共享关系(share relationship):如果两个挂载对象具有共享关系，那么一个挂载对象中的挂载事件会传播到另一个挂载对象，反之亦然。
  • 从属关系(slave relationship):如果两个挂载对象形成从属关系，那么一个挂载对象中的挂载事件会传播到另一个挂载对象，反之却不行；在这种关系中，从属对象是事件的接收者。

挂载状态：

  • 共享挂载(shared)
  • 从属挂载(slave)
  • 共享/从属挂载(shared and slave)
  • 私有挂载(private)
  • 不可绑定挂载(unbindable)

默认情况下，所有的挂载都是私有的。

User Namespace

User Namespace 主要隔离了安全相关的标识符和属性,包括用户ID、用户组Id、root目录、key以及特殊权限。比较常用的是，在宿主机上以一个非root用户运行创建一个User Namespace，然后再User Namespace中却映射成root用户。这意味着，这个进程再User Namespace中有root权限，但是在User Namespace外却没有root权限。从Linux Kernel 3.8开始，非root进程也可以创建User Namespace。

Network Namespace

Network Namespace 隔离的是和网络相关的资源，包括网络设备、路由表、防火墙(iptables)、socket（ss、netstat）、 /proc/net 目录、/sys/class/net 目录、网络端口(network interfaces)等等。

Network Namespace可以让每个容器拥有自己独立的(虚拟)网络设备，而且容器内可以绑定到自己的端口，每个Namespace内的端口都不会互相冲突。在宿主机上搭建网桥后，就能实现容器之间的通信。
