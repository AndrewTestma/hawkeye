Python-类继承
发表于 2021-02-26| 更新于 2021-03-03|Python
字数总计:2.8k|阅读时长: 10 分钟

　　面向对象编程 (OOP) 语言的一个主要功能就是“继承”。继承是指这样一种能力：它可以使用现有类的所有功能，并在无需重新编写原来的类的情况下对这些功能进行扩展。
　　通过继承创建的新类称为“子类”或“派生类”，被继承的类称为“基类”、“父类”或“超类”，继承的过程，就是从一般到特殊的过程。在某些 OOP 语言中，一个子类可以继承多个基类。但是一般情况下，一个子类只能有一个基类，要实现多重继承，可以通过多级继承来实现。

继承

在Python中，每个创建的类都使用了继承。所有的Python类都是object特殊类的子类。如果不显式的指明创建的类从其他类继承，那么这个类自动继承object为父类。

可以看出Person类默认继承了object，而显式指明继承的Women类继承了Person和object两个类。

__mro__可以列出类继承的所有“真实”超类，但是无法列出“虚拟”超类。

重写

父类的成员都会被子类继承，当父类中的某个方法不完全适用与子类时，就需要在子类中重写该方法。重写方法只需要保证子类与父类名称一致，不需要关系参数类型和个数。

Super

当子类中也定义了__init__()函数时，调用的父类的__init__()函数时主要有两种方式：

  • 直接使用父类名称
  • 使用super()函数 python中的super()方法的设计目的是用来解决多重继承时父类的查找问题，super()的好处就是可以避免直接使用父类的名字。

多态

python不支持多态，也不用支持多态。因为python本来就是多态语言，崇尚鸭子类型。

鸭子类型允许我们使用任何提供所需行为的对象，而不需要迫使它成为一个子类。

很明显的看出，Person类拥有跟Duck类一样的方法，当有一个函数调用Duck类，并利用到了两个方法walk()和swim()。传入Person类也一样可以运行，函数并不会检查对象的类型是不是Duck，只要他拥有walk()和swim()方法，就可以正确的被调用。

再举例，如果一个对象实现了__getitem__方法，那python的解释器就会把它当做一个collection，就可以在这个对象上使用切片，获取子项等方法；如果一个对象实现了__iter__和next方法，python就会认为它是一个iterator，就可以在这个对象上通过循环来获取各个子项。

由于python属于动态语言，当你定义了一个基类和基类中的方法，并编写几个继承该基类的子类时，由于python在定义变量时不指定变量的类型，而是由解释器根据变量内容推断变量类型的（也就是说变量的类型取决于所关联的对象），这就使得python的多态不像是c++或java中那样，定义一个基类类型变量而隐藏了具体子类的细节。

抽象基类

从Python2.6开始，标准库提供了抽象基类。大多数抽象基类在collerctions.abc模块中定义，不过其他地方也有。例如，numbers和io包中就有一些抽象基类。但是,collections.abc中的抽象基类最常用。Python3.4在collectons.abc模块中定义了16个抽象基类，可以从官方文档中查看各个抽象基类的描述和关系。

标准库中有两个名为abc的模块，这里说的是collections.abc。为了减少加载时间，Python3.4在collections包之外实现这个模块，因此要与collections分开导入。

声明抽象类最简单的方式是继承abc.ABC或其他抽象基类。如果使用的是旧版Python,则需要使用__metaclass__ = abc.ABCMeta声明抽象类：

除了@abstractmethod之外，abc模块还定义了@abstractclassmethod、@abstractstaticmethod 和
@abstractproperty三个装饰器。然而从Pyhon3.3起，后面撒啊个装饰器就废弃了，因为装饰器可以在@abstractmethod上堆叠，其他的就显得多余了。

在函数上堆叠装饰器的顺序通常很重要，@abstractmethod的官方文档特别指出了：与其他方法描述符一起使用时，abstractmethod()应该放在最里层。

特性

抽象类通常有几个特性：

  • 不能被直接实例化

  • 目的就是让别的类继承它并实现特定的抽象方法

  • 主要用途是在代码中检查某些类是否为特定类型，实现特定接口

  • 除了继承的方式，还可以通过注册方式来让某个类实现抽象基类

    当将一个类注册为抽象基类的虚拟子类时，使用issubclass、isinstance等函数都能识别，但是虚拟子类不会从抽象基类章继承任何方法和属性。

虚拟子类不会继承注册的抽象基类，而且任何时候都不会检查它是否符合抽象基类的接口，即便在实例化时也不检查。为了避免运行时错误，虚拟子类一般都需要实现所需的全部方法。

多重继承

实际编码中，可能会遇到父类派生出来许多子类。但父类的方法已经不能满足某个子类的需要，需要子类自行创建一个方法，这个方法也被其他的子类需要：

我们声明了一个Transportation代表交通工具类，并且有三个子类Car、Airplane和Helicopter。首先fly是不能写入到Transportation的父类中，因为汽车没有飞行的功能，只能在各自的子类中具体实现。但是这样的编码又违反了代码尽可能重用的原则。

那如果再声明一个Aircraft飞行器类，并让Airplane和Helicopter多重继承Aircraft, Transportation。这样的作法显然也违背了继承必须是IS-A关系。

在Java语言中实现多重继承的方式是采用interface。而在Python中，多重继承最简单、最有用的是Mixin模式。

Mixin模式

Mixin表示混入(mix-in),以***Mixin命名的类一般称为混入类,它代表了这个类是作为功能添加到子类中，而不是父类。
使用建混入类的注意事项：

  • 混入类不能直接被实例化使用
  • 混入类没有自己的状态信息，也就不需要定义__init__()方法
  • 混入类没有实例属性，通过__slots__ = ()来进行限制

所以混入类单独使用起来没有任何意义，它们是用来通过多继承来和其他映射对象混入使用的。实现混入类的方法有两种：

  • 多继承混入
  • 装饰器混入

子类化内置类型

在Python2.2之前，内置类型如list或者dict不能子类化。在Python2.2之后，内置类型可以子类化了，但是有个重要的注意事项：内置类型不会调用用户定义的类覆盖的特殊方法

可以看出，dict.update方法忽略了AnswerDict.__getitem__的返回值。
直接子类化内置类型容易出错，因为内置类型的方法通常会忽略用户覆盖的方法。不要子类化内置类型，用户自定义的类应该继承collections模块。

可以看出，子类化内置类型的问题只发生在C语言实现的内置类型内部的方法委托上，而且只影响直接继承内置类型的用户子doing一类。如果子类化使用Python编写得类，如UserDict或者MutableMapping,就不会受此影响。

方法解析顺序(MRO)

在支持多重继承的Python中，从父类中查找方法的顺序一般称做方法解析顺序(Method Resulution Order),即MRO。

Python发展至今，经历了一下3种MRO算法，分别是：

  • 从左往右，采用深度优先搜索(DFS)的算法，称为旧式类的MRO
  • 自Python2.2 版本，新式类在采用深度优先搜索算法的基础上，对其做了优化
  • 自Python2.3版本，对新式类采用了C3算法。由于Python3.x仅支持新式类，所以该版本只使用C3算法。

关于C3算法的详细解释可以参看Michele Simionato的论文The Python 2.3 Method Resolution Order。
